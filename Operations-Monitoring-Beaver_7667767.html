IT Documentation : Operations Monitoring Beaver

Created by Andi Asmara, last modified on apr. 01, 2022

Product Summary

A digital solution (desktop + mobile app) for clients to 24/7 monitor their dredging operation remotely. Key features: ​

         - Dashboard with KPI’s and alarms​

         - Overview of dredging project and conditions

Problem Summary

Background

Connected machinery becomes common. IHC offers integrated solutions to our customers. Digital services are one of the drivers of providing integrated solutions. ​The leading position of the IHC Beaver will be improved.

To achieve optimal use of a customer’s Beaver and increase their operational efficiency, we provide data driven intelligence. This provides insight, control and ease of mind, by asset and performance operations monitoring.  

1-sentence Goal

The goal of the Operations Monitoring module is to support our customers in monitoring the operational performance of their assets.

Solution Summary

  • Use Searchlight API to get company and vessel information

  • Use existing Ewon Cloud to get data from Vessel

  • Use TimescaleDB for the primary timeseries datastore

  • Use Azure Function to ingest timeseries data to save cost

  • Use existing Data Platform pipeline to get data from IFS

  • Use embedded Grafana in Mendix app to show dashboard

  • Use IBM Security Verify as Customer IAM

  • Use React-Native to wrap Mendix Web App as iOS and Android native app

Insights

The spec was intended to be rebuild the previous version of Remote Monitoring Mendix app (IRIS). However, many new features are introduced and the IHC in-house styling was also changed. The goal was updated to create a new Mendix app, that will be called as Operations Monitoring module.

The core value is the utilisation of Grafana that has direct connection with TimescaleDB which will simplify the development and flexibility of the solution.

The usage of Customer Identity and Access Management is needed for Single Sign-On between Mendix and Grafana

Scope Details

Input Constraints

  • Use existing architecture as much as possible

  • No or minimal changes in the ingestion pipelines:

  • Company and Vessel information from IFS

  • IoT data from Vessel to TimescaleDB

  • No or minimal changes to IHC APIs:

  • Searchlight API

  • IHC CIAM API

Important Technical Decision

ITD 1 - Use embedded Grafana in Mendix for dashboard

THE PROBLEM

Operations Monitoring module shows KPI and alarm in real time to the users. Data pipeline is already existed to convey the data from vessels to timeseries database in the cloud (TimescaleDB). There are around 250 signals (including alarms) that available to be shown as KPI and alarms.

The solution must give flexibility for the IHC administrator to create different KPI & alarm dashboard for different customers.

What solution that need to be used for this?

OPTIONS CONSIDERED
(Decision in bold)

  • Use Mendix dashboard functionality with data from IHC API

  • Use Mendix dashboard functionality with data copied from TimescaleDB to Mendix

  • Use embedded Grafana connected directly to TimescaleDB

REASONING

Getting data through IHC API means that specific API must be built for each KPI/Alarm. Means a lot of effort needed.

Copy the timeseries data to Mendix will unnecessarily data movement and data storage.

The best option is to use Grafana and embed it in Mendix app. Grafana dashboard has a good interface with TimescaleDB and only will query data that are necessary for the KPI in real-time.

FEEDBACK

To ensure that a user from one company only can see the dashboard of the company where the user works, both Mendix app and Grafana must be secured, and SSO must be implemented

ITD 2 – Use IBM Security Verify as Identity Provider for both Mendix and Grafana

THE PROBLEM

Operations Monitoring module contains embedded Grafana dashboard in Mendix app. For functionality and security reason, both Grafana and Mendix must be authenticated with the same IdP or multiple federated IdPs.

How should we secure the application?

OPTIONS CONSIDERED
(Decision in bold)

  • Store the user identity in both Mendix application database and in Grafana database and configure that both trust each other.

  • Use existing IHC Azure Active Directory.

  • Use IBM Security Verify for both Mendix and Grafana.

REASONING

Storing user entities in the application database ensures referential integrity of related entities in the database. However, using temporary credentials is a hack and should be avoided. It is preferred to store the users and authenticate dedicated Identity Provider. Since this is a customer facing application, the usage of IHC Azure Active Directory is not preferred.

ITD 3 - Store application data on Mendix

THE PROBLEM

Operations Monitoring module must store the application data such as customer’s company, assets, and projects, including some additional detail about the dredging situation.

Where should we store this data?

OPTIONS CONSIDERED
(Decision in bold)

  • Store on Azure SQL database and use IHC API to do CRUD.

  • Store on Mendix.

REASONING

By storing it in Mendix, the implementation of the WebApp will be much faster since the data model and storage can be easily implemented as standard in Mendix. Also, no additional API is required.

FEEDBACK

To be noted, when any data is needed by other application, or will be used for advances analytics, the standard ingestion pipeline must be used.

Topic: KPI & Alarm dashboard

Background

Users are stored in single IdP

Each Grafana environment contains multiple folders, each folder contains 1 or more dashboard

User access to the dashboards in a certain folder is limited to the users that are a member of a certain Team

Each company in Mendix application, has one-to-one relation with Team and Folder in Grafana

When a new company is provisioned in Mendix application, a new folder and a new Team are created in Grafana

When a user is provisioned in Mendix application, it also will be automatically provisioned in Grafana and put that user as a member in the related Grafana Team.

ITD 4 - Use Folder and Team is Grafana to control user access.

THE PROBLEM

Operations Monitoring module will use embedded Grafana in Mendix. For security, it is not enough only to control user access in Mendix app, because if a user knows the Grafana URL for other company’s dashboard, then it can be directly access. Thus, the user access must be used in Grafana as well.

How should we implement notification the user access in Grafana?

OPTIONS CONSIDERED
(Decision in bold)

  • Use access control in each dashboard.

  • Use Grafana Team that linked to Company in Mendix and give access to a dashboard.

  • Use Grafana Team that linked to Company in Mendix and give access to a Folder.

REASONING

Every company will have multiple dashboard, thus, it will be easier to maintain if the user access is applied to that company’s folder, not to individual dashboard.

FEEDBACK

The automatic provisioning between Mendix and Grafana must be implemented in a robust way.

Topic: Mobile Native app deployment to AppStore and Play Store

Background

ITD 5 – Use React-Native to wrap the Mendix WebApp

THE PROBLEM

Operations Monitoring module uses Mendix platform. Since there is no intention to make it available offline, it will be built as Mendix WebApp. However, mobile version is needed. To make the app distribution as wide and as professional as possible, both iOS and Android versions are needed and must be published in both AppStore and Play Store

How should the mobile app be created?

OPTIONS CONSIDERED
(Decision in bold)

  • Use Mendix Native App.

  • Use Progressive Web App

  • Wrap the Mendix WebApp as a Native app using React-Native.

REASONING

The simplest approach is to create a simple React-Native app.
