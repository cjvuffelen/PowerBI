Introduction

To work with power bi an understanding is needed for source control, deployments and refresh.

 

Source control

For Power BI a separate Git is created: https://github.com/Royal-IHC-BV/PowerBI to store the different items.

In order to work with source control for Power BI, you need the following:

  • Contributor rights on Git

  • Installation of Git desktop

  • Clear communication in the team on which part you’re working on, as the pbit files are binary, and therefore no part development is possible.

The following parts are stored in source control

  • Custom Visuals
    for every custom visual, store an example pbix file and a custom visual file.

  • Programming.
    (dax, power-shell, tsml…) Store all relevant scripts which are used to maintain the Power BI solution

  • Templates

    Standard datasets, pbit files for reporting templates etc.

  • Themes

    Themes which are custom created.

  • Work-spaces

    1. For every CDM maintained workspace a folder

    2. In every workspace a folder for:

      1. Reports

      2. Datasets

      3. Excel

Please keep in mind that files with the pbix extension are binary files and are therefore not to be merged. So files will be overwritten. Therefore development with multiple people on same topic should be done with great care.

Collaboration

As said, pbix files are binary, it’s not possible to have a merge conflict of any kind. therefore it’s important to stick to the way of working for CDM power bi development:

  1. Communicate what you start to develop and on which model to make sure no-body is touching that file.

  2. Get latest version using git desktop from master

  3. Start building in master.

  4. Manually deploy to test

  5. After testing commit your stuff and deploy to acceptance and prod

  6. Communicate the work is done.

 

Refreshes on data model

A power bi model can be refreshed in multiple ways

Manual refresh using sql server management studio

Via sql server management studio additional options are available to refresh power bi data sets manually. via this tool it is possible to only refresh a table, a set of tables or only recalculate to update the measures or meta data. this is very handy for fast development on measures and meta data only as you don’t need a full refresh.

how to:

  1. Install sql server mangement studio

  2. Find the name of the server in the power bi service by data set settings:

  3. Connect to analyses server using your name and password

  4. Once connected you can see the objects in the object explorer:

  5. Right click on a table or on central model to perform an action

  6. On process database you will have the option to recalculate only “Process Recalc”

  7. Click on script action to new query window, this will open a new query window

  8. execute the script

For more information on possibilities and in relation to large datasets please read advanced incremental refresh here.

Deployment

For the deployment in power bi three options are available:

  1. Manual deployment

  2. Using power bi deployment pipelines

  3. Using PowerShell own designed deployment

For central model and integrated project performance reporting the pipeline is created.

Using power bi deployment pipelines

please read content here to understand deployment pipelines in general.

Implementation of IHC example of one workspace:

  1. Created three workspaces using naming convention

    1. workspace name test

    2. workspace name acc

    3. workspace name

  2. Create a new deployment pipeline

  3. Assign created workspaces to stages dev / test / prod
    (Please keep in mind that the naming in power bi is different then in data bricks.)

  4. Publish from pbi desktop to test workspace

  5. Add deployment rules for data sets

  6. Use deployment pipeline to deploy to next stage

  7. optional: on production update the app via the deployment pipeline
