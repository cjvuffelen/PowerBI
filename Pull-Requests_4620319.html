IT Documentation : Pull Requests

Created by r.lodder1, last modified on mrt. 15, 2022

A Pull Request (PR) is an event where (a) developer(s) are asking for approval fore merging the changes to a different branch. In the case of Integrations, it is always a permission for a merge to the main branch. A Pull Request is more than just a permission for a merge, it is an event that triggers multiple steps in the background which all contribute to a more stable experience.

Approvals/ Codeowners

For MuleSoft Integrations, it is required that at least one person (other than the one who initiated it) gives an approval on the code. Instead of “hoping” that someone picks it up or “randomly assign it”, it is decided to do an auto-assignment based on review load.

Per repository it must be defined which group(s) is/are responsible for this repository. Based on this information, the setup of the group is used to define who the members are of this group. Then, the amount of person(s) are assigned to look into this Pull Request.

This step is relying on CODEOWNERS and GitHub setup as defined in Setup CI/CD in GitHub Team

Pull Request Checks

When creating a Pull Request, GitHub will look at PR Checks on Repository Setting and on Workflow level. Repository level checks are defined in the repository settings and contains checks like the amount of approvals needed to approve a PR and when a specific merge can actually be done (e.g. an x amount of Workflows need to be successful before merging).

The other checks made are defined in the ‘workflow’ directory, containing GitHub Workflows. These workflows are in place to execute the actual CI/CD part like compiling, testing and deploying and will execute after a PR, commit on a specific branch etc. The most important Workflow is the Main CI/CD workflow and covers all of these topics.

Because a Mule template must be created from the template, all checks are the same. The list can be found in Setup CI/CD in GitHub Team

Merge requirements

To guarantee that code can be compiled and is deployable, all the required GitHub Workflows need to be successful before a merge can be executed. If one of the required GitHub Workflows is failed, the merge cannot be executed and the developer who initiated the PR needs to fix this problem. An up-to-date list of checks can be found in the Setup CI/CD in GitHub Team
