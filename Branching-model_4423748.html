IT Documentation : Branching model

Created by r.lodder1, last modified on mrt. 15, 2022

For MuleSoft Integrations it is decided to make use of the GitHub Flow branching model. This branching model is known for its simple and straight forward branching model which is easy to implement and suits well for the use cases that Royal IHC has for Integrations, mainly because the integrations are all about small deployable assets and don’t have any underlying direct dependencies to other integrations.

Branches

GitHub Flow is a straight forward branching method that only requires two branches, the ‘main’ and a ‘working’ branch for development. The main branch is the stable branch that contains the last known state of deployable code. When a change (whether it’s a bug, feature or something else) needs to be made, a new branch is created from this main branch. Development takes place on this branch, including the pull requests and further discussions.

The top line is the main branch, the bottom line is the branch in which development takes place

As mentioned before, there is a line between the main and the ‘development’ branch. To fully support this, the Prebuild Github Workflows, repositories and platform are structured in such a way that for MuleSoft Integrations there is no room for own interpretations.

Main branch responsibilities

Defined below is a summary of the responsibilities. More detailed information can be found in the configuration of the components above.

  • The main branch always contains code that is deployable (i.e. successful compilation)

  • No commits can be done manually and/or by a GitHub personalized user

  • Main branch code only deploys to QA and Production

‘Development’ branch responsibilities

  • New development can only take place in a ‘development’ branch

  • A merge to the main branch is only possible via Pull Requests

  • The development branch only deploys to the DevTest environment
