Introduction

Power BI can be used in many ways in a company. From personal ad-hoc reporting in a local workspace to enterprise models in a centralised workspace, providing reports for hundreds of end users. From manual deployment (and development) to fully automated deployment and refresh pipelines. This development guideline and standard is intended for enterprise development. Power BI developers and report builders which create content for IHC can benefit from these standards and best practices. Topics described:

  • Data models

  • Reports

  • Custom visuals

  • Themes

The purpose is to increase uniformity, readability and maintainability on central data models, reports and layouts using standards and best practices.

This guide uses the development standard and guidelines of the dimensional model from the data warehouse as a basis. The development standard and guidelines are used in CDM to develop a unified standardized dimensional model for IHC reporting and analytics and affects the naming convention used in pbi_datamodel. A basic understanding of the data warehouse and corresponding data layers is needed to understand the full dept of this document.

Furthermore, Microsoft has published a white paper for Power BI enterprise deployment with a lot of information to learn from, which gives context to choices made in this document.

Some of best practices are copied from internet using websites of Power BI experienced influencers like https://www.sqlbi.com/

Disclaimer: An organisation can have perfectly described best practices and development guidelines without keeping them. Therefore, I would recommend to use these guidelines as part of the review and therefore as part of the definition of done in the scrum process.

Source control

Source control is used in order to keep history of all changes on the existing model. Source control is needed once a bigger team is responsible for development and maintenance of Power BI. The following parts should be kept in source control:

  • Reports

  • Data models

  • Themes

  • Custom visuals

  • Programming stuff (Dax, powershell, tabular editor, C#)

  • pictures (used in power bi)

In IHC we use Git as source control.

(see installation and setup document from CDM)

Dashboards are not part of Git as they only live in the service.

Transformations and calculations

Power BI is also strong in transforming data either using transformations or using Dax. On the other hand, there are other things to consider when designing the transformations. What if the size grows? What if additional cross checks are needed, what if the process needs to stop in case the transformation is wrong? Etc.

There are two options:

  1. Transform in a Data warehouse layer

  2. Transform in Power BI

There are several reasons to develop transformations (for enterprise usage) on the data warehouse side:

  • Re-usability on the Power BI side (loading same dimension in multiple Power BI models

  • Maintainability, control of pipeline and error handling is much easier at the DWH side, including full reloads and historical adjustments

  • One definition for all, definitions are predefined in transformation layer based on standardised business rules

on where which calculations are done. For enterprise development the standard is to have all transformations done in the data warehouse layers. Exceptions on this standard must be discussed in the team and documented.

In Power BI

  • Time related calculations
    year to date, last 5 days, etc.

  • Calculations including other measures
    variance = planned – actual

  • %progress = remaining / actual: load remaining and actual and calculate % progress. Avoid loading %progress directly.

Modelling standards

  

  • Language of tables, columns, measures etc in English

  • Dates

    • Create one centrally managed date table with all columns and attributes needed for reporting. (done in data warehouse)

    • Classify this table as date table

    • Disable the auto date calendars setting in Power BI desktop

    • Power BI can only handle one active relation with a dimension in one table. In case a fact has multiple date fields (sales date and delivery date) use one date dimension and multiple (indirect) relations to avoid confusion with end-user.

    • Use date as key in relation to date dimension instead of integer

    • No time in date fields (or with reason)

    • Pay special attention on weekly reporting by year end, force usage of standards like iso week numbering etc. IHC standard = ISO week numbering

    • First day of week is on Sunday

  • Time

    • The entire DWH is using one time zone (CET?) which result in consistent reporting.
      Additional investigation is needed for multi time zone reporting.

    • Create separate time dimension for intraday reporting

    • Use integer as key in relation to time dimension instead of datetime (separate time from date) and only applicable if reporting on time is needed.
      can we store only time in as key (1900 + time)?

  • Model setup

    • Test data
      Use parameters to filter and reduce dataset PBIX size to ~400 MB or less
      Or
      Create a test database with a limited set of consistent data

    • For datasets larger than 500 MB first evaluate consider the following options:

      • reduce the number of columns or rows

      • reduce the granularity without losing functionality

      • aggregation tables

    • Create incremental refresh policies for dataset larger than ~500 MB (compressed)

    • Snow flaking
      In most cases it’s not needed. Merge tables instead. For example, a product category table can easily be integrated in the product table having better performance and less storages.
      Snow flaking is used when bridge tables are needed to fix a many to many. (for example: one item must appear in multiple hierarchy at the same time)

    • Bi-directional relationship should be avoided wherever possible.

    • Hide all key columns and other columns not used directly by users, or remove columns from model

    • Load underlying content instead of pre-calculated values: For example, if % progress is calculated from remaining / actual, then load the remaining and actual fields and not the % progress.

  • Measures

    • Create explicit measures for all aggregate-able column values and hide aggregate-able columns unless they are needed for grouping

    • Set all non-aggregate-able numeric column default summarization to “Do Not Summarize”

  • Business modelling choices

    • Multi-currencies:

      • Avoid multiple currencies in one model if not required

      • Avoid Conversion of currency in powerbi, load pre-calculated values instead

      • Use base amount and euro amount to show both in one fact

Column formatting

Standards for formatting columns in the power bi model.  IHC has predefined set of formats to be used across the organization.

  • Format add whole numbers with thousand separators (1,000)

  • Format all currency & decimal measures to defined standard (perhaps 2 decimals, thousand separator) 1,000.52

  • Format all ratios and percentage with appropriate (one) decimal positions 15,2%

  • Format all dates to “YYYY-MMM-DD” as international users do not understand the Dutch “yyyy-mm-dd”.

Measures and calculated columns

  • Use the DIVIDE function for all division operations to avoid div by zero

  • Assign measures to fact table where they logically belong – or to an empty measure group table.

  • Measures that don’t have an obvious home (fact) table should be assigned a measure group table.

  • Avoid using Dax function: “allselected” if possible due to complex and unexpected behaviour in composite models
