Workspace intro

Introduction to work-spaces can be given by reading the usage scenarios chapter in the whitepaper planning a Power BI Enterprise Deployment. With several pictures Microsoft explains the concept of work-spaces for small teams till enterprise solutions. A workspace is a container for collaboration. From that perspective any initiative with collaboration on data and reporting can be grouped in a workspace.

Storing and viewing in power bi service.

Storing is only possible in two type of work-spaces: my personal, or an organisational workspace. Only on an organisational workspace an app can be created. An app is a way to distribute content to a large group of viewers.

Workspace considerations

Some more considerations about creating work-spaces. There is no golden rule. Creating a workspace per department is to broad, as only one app can be created on top of a workspace. To create a workspace for every collaboration (ad-hoc or permanent) will lead to chaos in workspace management.

Considerations by designing workspace structures: (some online content for further details)

  • Responsibility based
    If a different department is responsible for the reports then those who the content is delivered to, or if the data is delivered by a different department. Then it makes sense to have multiple work-spaces.

  • Content based
    What content is delivered.

    • Subject
      for example a company wide project management tool, or a project wide cost control app. Or a departmental process analysis.

    • Sensitivity of data
      For example: Strategic data should be separated from normal company data
      For example: Company critical, or high availability
      Which data is sensitive within the company and is the company open to share data?

    • Split data from reports to have better management on access.
      In case that not all users are allowed to change a central(ised) data model.

  • Audience based

    • Enterprise

      • Organisational structures

      • Functional discipline

      • Corporate discipline

      • Capacity Center

      • Product Market Group

      • Teams in one of above-mentioned groups

    • User level on any collaboration

  • Capacity based

    • Only pro users don’t need Premium

    • And the other way around only readers need premium

  • Environment based
    In case testing and accepting is needed. (test / acc / prod)

Please keep in mind, changing a workspace is easy in Power BI, but linked content from documentation or websites will stop working.

Workspace creation

There is no clear general rule available to choose the workspace name. Based on assumptions at IHC a first set of rules is prepared in order to guide those who are allowed to create work-spaces.

Assumptions in IHC workspace setup

  1. First gateway for access is the Power BI license. All licensed persons can see the data unless it is restricted

  2. Restricted data are

    1. personal sensitive data (Time registration, HR-related)

    2. Defense data

    3. Strategic and sensitive data. (Specific finance data)

  3. IHC wants to grow to a more enterprise mode setup

  4. IHC wants to limit the work-spaces to used and needed work-spaces

Naming conventions

There is no clear tree to select your workspace name. it all depends on content, ownership, audience etc. But some naming conventions are created to keep structure in the workspace setup:

  • Ownership, or responsibility or subject area

    • Enterprise (IHC prefixed)
      -> IHC time registration
      -> IHC central data

    • Corporate discipline (CD prefixed)

    • Functional discipline (FD prefixed)

    • PMG (PMG prefixed)

  • Use official names ? from labelling project (Ronald Schuil), Petko Petkov

  • Project specific (prefixed with project yyyy)
    -> project 2021 GRIP

  • Environment if applicable

    • Test: suffixed with “ test”

    • Acceptance: suffixed with “ acc”

    • Prod: no suffix, but original name

  • Purpose (data / analytics specific)
    -> IHC central data, a workspace only for data

  • Meaningful and descriptive with spaces

  • Abbreviations in capital

  • Names with capital

  • All the rest lower case

  • Split data from reports by workspace
    In case a data set is developed by a different person / department a separate workspace will avoid unnecessary rights on data objects. (suffixed for data)

Workspace settings

In the workspace settings the license mode can be selected, the capacity and some other options.

In general:

  • Capacity:

    • Premium for work-spaces with files exceeding 3 gb

    • Premium for work-spaces connected to deployment pipeline

    • Premium for work-spaces with read only users

    • Pro when only pro users are collaborating

  • Data storage

    • Large for work-spaces where large datasets are expected like IHC central data

    • Normal for the rest.

  • Contact list:

    • Workspace admins

Furthermore, always assign at least two workspace admins, so those two are responsible for content etc.

Work-spaces and access rights

  • Work-spaces are only to be created by CDM or admin user

  • At least one admin must be assigned to a workspace
